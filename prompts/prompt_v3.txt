# {template_id} v{template_version} SMT constraint extraction — prompt v3

You are extracting structural constraints from the Asset Administration Shell
Submodel Template specification {template_id} {template_name} v{template_version}.

NEW IN v3: honesty, abstention, and calibrated formalization_status.

PRIMARY OBJECTIVE

This research evaluates whether constraints in the SMT PDF can be reliably
extracted and reliably distinguished from constraints that cannot. The goal is
not to maximize the number of records. The goal is to emit atomic constraints
whose status is honestly classified.

Over-formalization, such as calling ambiguous prose "exact", is a failure.
Under-formalization, such as correctly marking a rule "ambiguous" or
"prose_only", is success.
Honesty applies when a careful reading is genuinely uncertain. For
unambiguous table cells such as cardinality "1" or valueType "xs:string" in a
properties table, exact remains the correct status.

FIELD AND ATOMICITY CONTRACT

- field_idShort is required and non-empty. Use the idShort from the graph.
- aas_path is required and must exactly match a graph aas_path.
- For list children, use paths with "*", e.g. /{root_idshort}/Markings/*/MarkingName.
- Do not output field_idShort=null.
- Emit one atomic constraint per record.
- Split combined values: "xs:string [0..1]" is two records, not one.
- If one source statement could fit two classes, emit one record per class only
  when the values differ. Otherwise pick the most specific class, e.g.
  language_rule over cardinality, format over allowed_values.
  For instance, "the value shall be provided in at least one language" is a
  language_rule, not a meaningful cardinality rule on the field itself.
- Use canonical values only:
  - sme_type: Property, MultiLanguageProperty, File, SubmodelElementCollection,
    SubmodelElementList, Range, Blob, ReferenceElement, RelationshipElement,
    Entity, AnnotatedRelationshipElement, BasicEventElement, Operation,
    Capability.
  - value_type: xs:string, xs:boolean, xs:int, xs:integer, xs:long, xs:double,
    xs:float, xs:date, xs:dateTime, xs:gYear, xs:gYearMonth, xs:anyURI,
    xs:base64Binary.
  - cardinality: "1", "0..1", "0..*".
- Convert aliases before output: MLP, SMC, SML, String are not valid output
  values.

COMPACT UML / TABLE-LIKE BLOCK RULES

Some PyMuPDF paragraph blocks contain many compact UML rows inside ONE
evidence block, for example:

  +ManufacturerProductRoot : MLP [0..1]
  +CompanyLogo : File [0..1]
  +Markings : SML [0..1]

For this pattern:

1. The evidence_id belongs to the whole block. Every record derived from any
   line inside that block must cite that same evidence_id. Do not invent
   nearby block IDs for individual lines.
2. The text before ":" is the source idShort after removing "+".
3. MLP, SMC, SML, File, and Property after ":" are sme_type values.
4. xs:* values after ":" are value_type values.
5. Bracketed values such as [0..1] or [0..*] are cardinality values. Copy the
   inside of the brackets exactly: [0..1] -> "0..1", not "1".
6. If a compact row has no bracketed cardinality, do not infer cardinality "1".
7. Exact compact rows are not ambiguous merely because they come from a
   compressed paragraph block.

METADATA LABEL RULE

Labels such as "idShort:", "Class:", "semanticId:", "Parent:", and
"Explanation:" describe the current element. They are not field names. Never
emit field_idShort values such as "Class", "semanticId", "Parent", or
"Explanation". If the current element can be identified from nearby metadata
or the graph, attach the constraint to that element; otherwise skip it rather
than inventing a field.

CONSTRAINT_VALUE TYPE BY CLASS

Use the schema shape that matches the constraint_class:

- String values:
  cardinality, sme_type, semantic_identity, value_type, external_reference.
- Array of strings:
  allowed_values.
- Object values with required keys:
  structural_containment: {"parent_path": "...", "child_path": "..."}
  naming: {"rule": "..."}
  language_rule: {"min_languages": 1}
  conditional: {"antecedent": "...", "consequent": "..."}
  recommendation_vs_hard: {"statement": "..."}
  format: {"description": "..."}
  ambiguous: {"reason": "..."}

HOW TO PICK formalization_status

exact:
  Literal, unambiguous source value or rule, such as a table cell containing
  "0..1" for a named field.

approximate:
  Meaning is clear but the formal encoding loses nuance.

ambiguous:
  A careful reader could read the block more than one way, context is missing,
  or the block contradicts another block.

prose_only:
  The source states a real rule in prose, but it does not fit the current
  schema/taxonomy.

requires_external_resolution:
  The rule depends on an external resource such as ECLASS, IEC CDD, IEC, ISO,
  or another referenced standard.

HOW TO PICK normativity

hard_rule: "shall", "must", "is required to", or a normative table row.
recommendation: "should", "is recommended", "ought to".
example: "e.g.", "for example", "such as".
informative: "may", "can", or descriptive text without obligation.

If there is no modal verb in the immediate text but the table heading or
section heading establishes normativity, apply that section-level normativity.
Set formalization_status="ambiguous" only when neither inline language nor
section/table context resolves the rule's force.

OUTPUT FORMAT

Output a JSON array. No prose. No markdown fences.

Your response must begin immediately with "[" and end immediately with "]".
Anything before "[" or after "]" is invalid output.

{
  "field_idShort":        string, non-empty,
  "aas_path":             string, must match the graph,
  "constraint_class":     one of {cardinality, sme_type, semantic_identity,
                                  value_type, structural_containment, naming,
                                  language_rule, conditional,
                                  recommendation_vs_hard, external_reference,
                                  allowed_values, format, ambiguous},
  "constraint_value":     string or object, depending on constraint_class,
  "normativity":          one of {hard_rule, recommendation, example, informative},
  "evidence_ids":         array of input evidence_id strings, non-empty,
  "formalization_status": one of {exact, approximate, ambiguous, prose_only,
                                  requires_external_resolution}
}

SELF-CHECK BEFORE EMITTING EACH RECORD

1. Does aas_path resolve to exactly one graph node?
2. Is field_idShort non-empty and consistent with that node?
3. Is constraint_value atomic and canonical?
4. Does every evidence_id appear in the input?
5. If a block is too complex to atomize fully, did you emit only the atomic
   constraints you can cite confidently plus, if needed, one ambiguous record
   naming the complexity?
6. Is the confidence status honest?
