# {template_id} v{template_version} SMT constraint extraction — prompt v0

You are extracting structural constraints from an Asset Administration Shell
Submodel Template ({template_id} {template_name} v{template_version}) PDF.

This is the structured baseline prompt. It intentionally keeps the formal
record contract so the baseline failure modes are about extraction,
formalization, citation, and schema discipline rather than raw JSON shape.

Output a JSON array. No prose. No markdown fences. If you cannot identify any
constraint from the evidence blocks, output [].

Each JSON object must contain:
{
  "field_idShort":        string, non-empty,
  "constraint_class":     one of {cardinality, sme_type, semantic_identity,
                                  value_type, structural_containment, naming,
                                  language_rule, conditional,
                                  recommendation_vs_hard, external_reference,
                                  allowed_values, format, ambiguous},
  "constraint_value":     string or object, depending on constraint_class,
  "normativity":          one of {hard_rule, recommendation, example, informative},
  "evidence_ids":         array of evidence_id strings from the input, non-empty,
  "formalization_status": one of {exact, approximate, ambiguous, prose_only,
                                  requires_external_resolution}
}

Atomicity and canonical values:

- Emit one record per atomic constraint. Do not combine type and cardinality.
  Wrong: {"constraint_class": "value_type", "constraint_value": "xs:string [0..1]"}
  Right: one value_type record with "xs:string" and one cardinality record with
  "0..1".
- For value_type use only: xs:string, xs:boolean, xs:int, xs:integer, xs:long,
  xs:double, xs:float, xs:date, xs:dateTime, xs:gYear, xs:gYearMonth,
  xs:anyURI, xs:base64Binary.
- For sme_type use only: Property, MultiLanguageProperty, File,
  SubmodelElementCollection, SubmodelElementList, Range, Blob,
  ReferenceElement, RelationshipElement, Entity, AnnotatedRelationshipElement,
  BasicEventElement, Operation, Capability.
- Convert table abbreviations before output: MLP -> MultiLanguageProperty,
  SMC -> SubmodelElementCollection, SML -> SubmodelElementList.
- For cardinality use only strings like "1", "0..1", "0..*".
- For semantic_identity output only an IRI or IRDI literally present in the
  source. Do not use descriptions as semantic identities.
- Modal verbs map to normativity: "shall"/"must" = hard_rule;
  "should"/"recommended" = recommendation; "may" = informative or
  recommendation depending on context; "e.g."/"for example" = example.
- Never output recommendation_vs_hard as normativity. recommendation_vs_hard
  is a constraint_class value only.

If a block is ambiguous, prefer constraint_class="ambiguous" and
formalization_status="ambiguous" over inventing structure.
